# web2md 下一阶段方案：原文样式 1:1 还原 + 视频/音频导出

> 目标：让导出的 HTML 与微信原文「视觉等价」，并把当前被丢弃的视频/音频也带出来。
> 本文是**思路收集与设计草案**，不是最终实现规范，落地时再按章节拆 PR。

---

## 0. TL;DR

- **样式还原**走两条线并用：一是把微信外链 CSS 一次性拉下并内联进 `<head>`，二是补一段我们自己写的「微信原文等价基线」CSS 兜底未命中的规则。最终容器仍维持 `max-width: 677px`，body 背景对齐 WeChat webview（浅色 `#f7f7f7`，文章卡片 `#fff`）。
- **媒体支持**新增独立的 `media.py` 模块，专门识别 `<iframe class="video_iframe">`、`<mp-common-mpvideo>`、`<mpvideo>`、`<mpvoice>`、`<qqmusic>` 等微信私有标签，分流到「外链占位」「下载到本地」两种策略，HTML 渲染 `<video controls>` / `<audio controls>`，Markdown 用 `![]()` + 文字链接组合 + HTML 兜底块。
- **回退/容错**优先：任何媒体处理失败都不影响图文主线，只在 frontmatter 或 `_debug/` 里留痕。

## 1. 现状盘点

### 1.1 样式现状

`templates.py` + `common.extract_head_styles` 当前做法：

1. 自带一小段「外壳 CSS」：`body` 字体栈、`color:#222`、`line-height:1.6`、`.page` 容器 677px 居中白底。
2. 把微信 SSR HTML 里 `<head><style>...</style></head>` 全部 inline 抓过来，原样塞进输出。
3. **不抓** `<link rel="stylesheet">`——微信关键样式（`page_mp_article_improve_combo.css` 等）就在那里。

实测一个普通公众号文章导出后，`<head>` 里从微信带过来的内联 CSS 通常只有几百字节（如 `.cooldown_tips` 这类边缘类），核心排版规则**几乎都丢了**。所以现在看起来「像」微信，但细节都靠我们外壳 CSS 在硬撑：

| 项目 | 微信原文 | 当前导出 | 差异 |
|------|---------|---------|------|
| 页面背景 | `#f7f7f7`（webview 灰底） | 浏览器默认白 | 缺 |
| 文章卡片背景 | `#fff` | `#fff`（.page） | OK |
| 正文字号 | 17px（标准档，用户可调） | 浏览器默认 16px | 偏小 |
| 行高 | 1.75 | 1.6 | 偏紧 |
| 段间距 | `margin: 1em 0`（实测更松） | 浏览器默认 | 偏紧 |
| 文字色 | `rgba(0,0,0,0.9)` ≈ `#191919` | `#222` | 接近，差一档 |
| 链接色 | `#576b95` | `#576b95`（仅 meta） | 正文 a 未覆盖 |
| 引用块 | 左侧灰条 + 灰底 | 浏览器默认 | 缺 |
| 代码块 | 浅灰底 + 等宽 | 浏览器默认 | 缺 |
| 标题层级 | h1/h2/h3 字号阶梯 | 浏览器默认 | 差异明显 |
| 暗黑模式 | `data-darkmode-bgcolor` 等属性切换 | 不支持 | 缺 |

### 1.2 媒体现状

`parser_wx.rewrite_content` 当前会**直接 `.decompose()` 掉 `<iframe>`**，这意味着所有视频都没了。`<mpvoice>` / `<qqmusic>` 这种自定义标签虽然没被显式删除，但因为浏览器不识别又没有 fallback，等于「保留了空标签，画面上是空白一片」。

需要分两类处理：

- **视频**：腾讯视频（`v.qq.com`）+ 微信视频号（`mp.weixin.qq.com/mp/videoplayer`）+ 微信内嵌短视频（`mpvideosnap`、`mp-common-mpvideo` Web Component）。
- **音频**：`<mpvoice>`（公众号语音卡片）+ `<qqmusic>`（QQ 音乐卡片）。

## 2. 微信文章样式还原方案

### 2.1 策略 A：抓取并内联微信外链 CSS（推荐为主）

**思路**：解析 `<link rel="stylesheet" href="...">`，对其中宿主名匹配 `res.wx.qq.com` / `mp.weixin.qq.com` 的链接，发起 GET，把 CSS 文本写进 `<style>` 注入到 `<head>`，并把所有 `url(...)` 引用改写成绝对 URL（字体、icon 还是走 CDN，离线时降级）。

**优点**：拿到的就是「写文章那一刻」微信真正用的全部规则；不需要我们手动维护对照表。

**实现要点**：
- 在 `net.py` 增加 `fetch_css(url, client) -> str`，复用同一个 `httpx.Client`。
- 在 `common.extract_head_styles` 里新增 `include_external: bool = True` 参数；为 True 时遍历 `head.find_all("link", rel="stylesheet")` 并并发拉取。
- 命中白名单（`res.wx.qq.com`、`mp.weixin.qq.com`、`mmbiz.qpic.cn`）才拉，避免误抓第三方追踪脚本注入的 CSS。
- CSS 内的 `url(...)` 用一个简单的 `re.sub` 转绝对地址；不做内联（base64 化太重，且对 woff 字体无收益）。
- 失败静默降级：拿不到就跳过，不影响主流程；`--debug` 时记录到 `_debug/css_failed.txt`。

**风险**：
- 微信 CSS 体积可能 50–200KB，会让单篇 HTML 膨胀。可加 `--no-inline-css` 关掉。
- CSS 内部规则可能依赖 `<script>` 注入的 class（比如 `.appmsg_skin_box`），删 script 后这些 class 仍在 DOM 上是 OK 的。
- 个别规则使用 `:has()`、`@container` 等较新特性，老浏览器降级即可。

### 2.2 策略 B：手写「微信原文等价基线」CSS（推荐作为兜底）

**思路**：把社区已知的微信文章核心样式归纳成一份 `wx_baseline.css`，作为 `templates.py` 里的硬编码字符串，**在策略 A 抓取的 CSS 之前**注入——这样外链 CSS 会自然覆盖我们的兜底值，没抓到时也有合理外观。

### 2.3 容器与字号档位

微信 App 里用户能在「设置 → 字体大小」切五档（小/标/大/超大/极大），webview 通过 `font-size` rem 基准切换实现。我们简化：

- 默认渲染「标准档」(17px) — 与微信默认一致。
- 在 `templates.HTML_TEMPLATE` 里加个右上角小工具条（纯 CSS + 一段最小 JS，或干脆纯 `<a href>` 链接切换 class），支持 S / M / L 三档：

```html
<div class="font-size-toggle">
  <a data-size="s">小</a><a data-size="m" class="active">标</a><a data-size="l">大</a>
</div>
```

```css
.size-s .rich_media_content { font-size: 15px; }
.size-m .rich_media_content { font-size: 17px; }
.size-l .rich_media_content { font-size: 19px; }
```

> 此功能非必需，先做核心样式还原；字号切换列为 P2。

### 2.4 暗黑模式（P2，可选）

微信文章在元素上挂 `data-darkmode-bgcolor-xxxx`、`data-darkmode-color-xxxx` 属性，App 在 dark 模式下用 JS 把这些值搬到 `style`。离线还原方案：

- 注入一段 JS：在 `prefers-color-scheme: dark` 且文档刚加载完时，遍历带这些属性的节点，把值写进 `style.backgroundColor` / `style.color`。
- 或更安全的方式：写 CSS `@media (prefers-color-scheme: dark) { ... }`，提供一套「足够能看」的暗色基线（背景 `#1f1f1f`、文字 `#d4d4d4`、引用块 `#2a2a2a`），不追求像素级还原。

## 3. 视频导出方案

### 3.1 微信视频的几种载体

| 形态 | DOM 标记 | URL 特征 | 能否直链播放 |
|------|---------|----------|--------------|
| 腾讯视频嵌入 | `<iframe class="video_iframe" data-src="https://v.qq.com/iframe/preview.html?vid=XXX...">` | `v.qq.com` | 可（嵌 iframe 即可，需联网） |
| 视频号 / 公众号原生视频 | `<iframe class="video_iframe" data-mpvid="wxv_XXXX">` 或 `<mp-common-mpvideo vid="wxv_XXXX">` | `mp.weixin.qq.com/mp/videoplayer?...vid=wxv_XXX` | 拿到真实 mp4 需多一步接口（见下） |
| 短视频卡片 | `<mpvideosnap>` 自定义标签 | 同上 | 同上 |

#### 拿到真实 mp4 的接口

公众号原生视频的真实播放地址可以通过：

```
GET https://mp.weixin.qq.com/mp/videoplayer?action=get_mp_video_play_url
    &preview=0&__biz=<BIZ>&mid=<MID>&idx=<IDX>&vid=<WXV_ID>&f=json
```

返回 JSON 里 `url_info[*].url` 即可直接 GET 下载 mp4。**注意**：
- 需要 Referer = `https://mp.weixin.qq.com/`；
- URL 含签名，几小时后会过期（所以下载要在抓文章的同一次会话里完成）；
- 一些视频是 m3u8 切片，需要额外解析（首期跳过，只支持单文件 mp4）。

### 3.2 处理策略（两档可选，默认 B）

**A. 纯占位（最轻量）**
不下载，只在 HTML 中保留 iframe（指向 `v.qq.com` 或微信视频播放页），Markdown 中转成「📺 视频：[在原文中查看](https://...)」一行。优点是零网络成本；缺点是离线/原文删除后视频也没了。

**B. 选择性下载**（推荐）
- 腾讯视频：**只保留 iframe**（v.qq.com 链接长期有效，没必要下）+ Markdown 文字链。
- 公众号原生视频：调上面那个接口拿 mp4 URL，下载到 `videos/<sha1>.mp4`，HTML 改用 `<video controls preload="metadata">`，Markdown 用 HTML 块包裹 `<video>`（GFM 支持原生 HTML）。
- 失败兜底：转 A 方案。

### 3.3 输出形态

**HTML**：

```html
<!-- 公众号原生视频:已下载本地 -->
<div class="wx-video">
  <video controls preload="metadata" poster="videos/poster_xxx.jpg">
    <source src="videos/abc123.mp4" type="video/mp4">
    <a href="https://mp.weixin.qq.com/...vid=wxv_xxx">原文视频链接</a>
  </video>
</div>

<!-- 腾讯视频:保留 iframe -->
<div class="wx-video">
  <iframe src="https://v.qq.com/iframe/preview.html?vid=xxx&amp;auto=0"
          frameborder="0" allowfullscreen></iframe>
</div>
```

**Markdown**：

```markdown
<!-- 公众号原生视频 -->
<video controls src="videos/abc123.mp4" poster="videos/poster_xxx.jpg" width="100%"></video>

📺 视频原链：<https://mp.weixin.qq.com/...vid=wxv_xxx>

<!-- 腾讯视频 -->
📺 腾讯视频：<https://v.qq.com/x/cover/xxx.html>
```

> Markdown 里塞 `<video>` HTML 块在 Obsidian / Typora / GitHub 都能渲染；纯文本笔记软件（如 1Writer）会显示成代码——加一行文字链兜底。

### 3.4 实现位置

| 改动 | 文件 | 说明 |
|------|------|------|
| 新增 `web2md/media.py` | 新文件 | 识别、分类、下载、改写 |
| `core.process_article` 新增 `videos/` 目录 + 调用 `media.handle_videos` | 修改 | |
| `parser_wx.rewrite_content` 移除 `iframe` 的无条件 `decompose()` | 修改 | 改为「先 delegate 给 media，再删剩余 iframe」 |

## 4. 音频导出方案

### 4.1 DOM 结构

```html
<mpvoice
    voice_encode_fileid="MzA....."
    name="第三集 我们怎么走到今天"
    play_length="623000"
    low_size="2436.7"
    source_size="9747.0"
    high_size="9747.0"
    frameborder="0"
    class="js_uneditable custom_select_card weapp_card_voice"></mpvoice>
```

下载地址（音频 mediaId 直接拼）：

```
https://res.wx.qq.com/voice/getvoice?mediaid=<voice_encode_fileid>
```

需带 Referer = `https://mp.weixin.qq.com/`，返回二进制。**坑**：

- 大部分公众号音频是 `mp3` 或 `m4a`，能直接 `<audio>` 播放；
- **少量是 SILK 格式**（微信语音通话同款编码），浏览器无法直接播放，需要先用 `ffmpeg` 或 `silk-v3-decoder` 转 mp3。**首期不处理 SILK**，只下载原始文件，在文件名留 `.silk` 提示用户。

判断方式：拿到字节后看魔数（前 10 字节包含 `#!SILK_V3`）。

### 4.2 输出形态

**HTML**：

```html
<div class="wx-audio">
  <p class="wx-audio-title">🎵 第三集 我们怎么走到今天 （10:23）</p>
  <audio controls preload="none">
    <source src="audio/abcd1234.mp3" type="audio/mpeg">
    <a href="audio/abcd1234.mp3">下载音频</a>
  </audio>
</div>
```

**Markdown**：

```markdown
🎵 **第三集 我们怎么走到今天** （10:23）

<audio controls src="audio/abcd1234.mp3"></audio>

[下载音频](audio/abcd1234.mp3)
```

### 4.3 QQ 音乐卡片

`<qqmusic>` 涉及版权，**不下载**，只渲染成卡片状链接：

```html
<a class="wx-qqmusic-card" href="https://y.qq.com/n/yqq/song/<songmid>.html" target="_blank">
  🎵 QQ 音乐：<song_name> — <singer>
</a>
```

Markdown 中直接渲染为一行 link。

## 5. Markdown 转换链路的配合调整

`common.to_markdown` 当前调用 `markdownify(strip=["span"])`。视频/音频以 HTML 块出现时，要避免 markdownify 把里面的属性吃掉。两种做法：

1. **占位法**：`media.handle_*` 在 DOM 里把媒体节点替换成 `<div class="wx-media-placeholder" data-md="...base64..."></div>`，markdownify 转换完之后我们再做一遍字符串替换还原。
2. **白名单法**：给 markdownify 传 `convert=[...]`（或继承 `MarkdownConverter` 重写 `convert_video`/`convert_audio`），让它对 `video`/`audio`/`iframe` 走「原样保留 HTML」分支。

推荐 **2**，因为更稳；占位法依赖 base64 反序列化容易出 bug。

## 6. 配置项与默认值

| CLI 开关 | 默认 | 作用 |
|----------|------|------|
| `--no-inline-css` | off | 关闭抓取微信外链 CSS（输出更小，样式还原退化） |
| `--no-videos` | off | 不处理视频（最快，等同当前行为） |
| `--no-audio` | off | 不处理音频 |
| `--video-mode {iframe,download,both}` | `both` | iframe 只保留外链播放；download 强制下载所有可下载视频 |
| `--media-workers N` | `4` | 媒体下载并发数（独立于图片 `-t`） |

`pyproject.toml` 无新增依赖。SILK 解码若做就是 P2，引入 `pysilk` 或 shell 调 ffmpeg。

## 7. 风险与回退

- **微信 API 变更**：视频接口的 `action=get_mp_video_play_url` 字段可能改名。媒体处理失败必须 try/except 包住，降级为 iframe 占位 + 警告日志。
- **CSS 体积膨胀**：抓全部外链 CSS 后单篇 HTML 从 ~50KB 涨到 ~250KB。可接受，离线场景不在乎，且我们的图片本来就远比这大。
- **markdownify 升级**：白名单写法依赖 markdownify ≥ 0.11 的 `convert` 参数行为，pin 一下版本。
- **`<iframe>` 安全性**：当前删 iframe 是「为了安全 + 离线」，恢复后要确保 iframe `src` 只允许 `v.qq.com` / `mp.weixin.qq.com` 白名单域，杜绝注入。
- **大文件下载**：视频可能 50–500MB。引入 `--max-video-size MB`（默认 200），超过则只保留 iframe + 在 `_debug/skipped.txt` 留记录。

## 10. 待回答的开放问题

1. **样式还原优先级**：策略 A（抓外链 CSS）和策略 B（写基线 CSS）都做，还是只做基线？外链 CSS 引入了「跟随微信更新而漂移」的风险，但还原度最高。**建议**：默认两个都做，外链失败不致命。
2. **视频下载默认是否开启**？默认开会让首次跑慢一截，且对带宽敏感的用户不友好。**建议**：默认 `iframe` 模式（只保留链接），`--video-mode download` 显式开启下载。
3. **音频 SILK 解码**是否值得引入 ffmpeg 依赖？影响打包成单文件 .exe 的体积。**建议**：先不引入，留警告让用户自行转码。
4. **是否新增 `audio.md` 这类辅助清单文件**列出本文所有音视频资源？方便用户做媒体库索引。**建议**：在 frontmatter 加 `media:` 字段列出本地路径，不另开文件。
